Любой ключ конфигурации можно задать тремя способами, в порядке возрастания приоритета:

1. Файл конфигурации YAML (`config.yaml`).
2. База данных (таблица `config_values`) при запуске без файла конфигурации.
3. **Переменные окружения** (всегда побеждают). Имя переменной - это YAML‑путь, записанный
   заглавными буквами с заменой точек на подчёркивания - например,
   `ai.content_moderation.enabled` → `AI_CONTENT_MODERATION_ENABLED`. Выполните
  `./gennadium -export-env`, чтобы вывести полный список для вашей текущей конфигурации.

Значение каждого ключа см. в [config_ru.md](config_ru.md).

## Предварительные требования

- **Токен Telegram‑бота** от [@BotFather](https://t.me/BotFather) - обязателен.
- Бот должен быть **добавлен в ваш чат как администратор** (с правами удаления/ограничения).
- Для **чатов с темами (форумов)** запишите ID сообщений тем/тредов, которые хотите
  модерировать.
- **Go 1.26+**, если собираете из исходников (не требуется для Docker).
- *Опционально:* ключи Azure OpenAI / OpenAI‑совместимого API для функций ИИ; Azure Vision или
  OCR.space для модерации изображений.

> **Совет - как найти ID чатов:** добавьте бота в чат, отправьте сообщение и включите
> `debug.debug_telegram: true`, чтобы увидеть ID чатов и тем в логах. ID групп/супергрупп
> отрицательные (например, `-1001234567890`).

> **Совет - как найти ID темы:** в чате-форуме откройте сообщение в нужной теме, скопируйте
> ссылку на него (правый клик / долгое нажатие → *Копировать ссылку на сообщение*). Она
> выглядит как `https://t.me/c/1234567890/55/1001` - **среднее число** (`55`) - это ID темы.
> Для публичных чатов ссылка имеет вид `https://t.me/yourchat/55/1001`, ID темы в той же
> позиции.

## Локальная установка (бинарник)

```bash
git clone https://github.com/noiseonwires/gennady.git
cd gennady

# Создать конфигурацию из шаблона
cp config.example.yaml config.yaml
# Отредактируйте config.yaml: bot_token, admin.chat_id, moderation.chat_id и ключи ИИ

# Собрать
go build -o gennadium .

# Запустить (использует ./config.yaml и локальную базу SQLite в db/moderation.db)
./gennadium

# Или указать произвольную конфигурацию
./gennadium -config /etc/gennadium/config.yaml
```

Для минимальной рабочей конфигурации нужны `bot_token`, `admin.chat_id` и `moderation.chat_id`.
Это запускается в режиме **long‑polling** (входящие порты не требуются) с локальной базой
SQLite.

### Сборка с информацией о версии

```bash
go build -ldflags="-X 'main.version=1.0.0' \
  -X 'main.gitCommit=$(git rev-parse HEAD)' \
  -X 'main.buildTime=$(date -u +%Y-%m-%dT%H:%M:%SZ)'" \
  -o gennadium .

./gennadium -version
```

## Локальный systemd‑сервис

Для постоянно работающего Linux‑хоста установите бот как systemd‑сервис:

```bash
# Сначала соберите, затем:
sudo ./install.sh
```

Скрипт создаёт системного пользователя `telegram-bot`, копирует бинарник и конфигурацию в
`/opt/gennadium` и устанавливает unit `gennadium.service`. Затем:

```bash
sudo systemctl start gennadium
sudo systemctl status gennadium
sudo journalctl -u gennadium -f      # следить за логами
```

> Задайте переменную окружения `TZ` в unit‑файле сервиса (например,
> `Environment=TZ=Europe/Prague`), чтобы запланированные события (утреннее приветствие,
> ежедневная сводка, RSS) срабатывали в правильное местное время.

## Docker - локально, без веб‑интерфейса

Запустите бота в контейнере с long‑polling и смонтированными конфигурацией + базой данных.
Открывать порты не нужно.

**1. Подготовьте файлы** в рабочем каталоге:

```
config.yaml      # ваша конфигурация (web_ui.enabled: false, webhook.enabled: false)
db/              # каталог для базы SQLite (будет создан)
```

**2. `docker-compose.yml`:**

```yaml
services:
  gennadium:
    image: ghcr.io/noiseonwires/gennady:latest
    container_name: gennadium
    restart: unless-stopped
    environment:
      - TZ=Europe/Prague
    volumes:
      - ./db:/app/db:rw                       # постоянная база данных
      - ./config.yaml:/app/config.yaml:ro     # конфигурация (только для чтения)
```

**3. Запустите:**

```bash
docker compose up -d
docker compose logs -f
```

## Docker - локально, с веб‑интерфейсом

То же, что выше, но включите веб‑интерфейс и опубликуйте его порт, чтобы настраивать и
модерировать из браузера.

**1. В `config.yaml`:**

```yaml
server:
  listen_addr: "0.0.0.0"
  listen_port: 8080

web_ui:
  enabled: true
  path_prefix: "/manage-7f3a"   # измените со стандартного "/admin" на свой, трудноугадываемый
  password: "выберите-надёжный-пароль"
  otp_enabled: true          # также требовать Telegram‑OTP код (нужен super_admin_user_id)

admin:
  super_admin_user_id: 123456789   # ваш Telegram user ID - получает OTP‑коды
```

**2. `docker-compose.yml`** - опубликуйте порт:

```yaml
services:
  gennadium:
    image: ghcr.io/noiseonwires/gennady:latest
    container_name: gennadium
    restart: unless-stopped
    environment:
      - TZ=Europe/Prague
    ports:
      - "8080:8080"
    volumes:
      - ./db:/app/db:rw
      - ./config.yaml:/app/config.yaml:ro
```

**3. Запустите и откройте** `http://localhost:8080/manage-7f3a`.

> Веб‑интерфейс защищён паролем и (рекомендуется) одноразовым Telegram‑кодом. Если вы открываете
> его в интернет, поставьте перед ним HTTPS (например, обратный прокси) и оставьте `otp_enabled: true`.
> Также измените `web_ui.path_prefix` со стандартного `/admin` на свой вариант - так панель
> сложнее обнаружить автоматическим сканерам.

> **Модераторы?** Чтобы дополнительно выдать модераторам изолированную ограниченную панель,
> задайте `web_ui.public_url` (например, `http://localhost:8080` локально или ваш публичный
> HTTPS‑URL) и оставьте `web_ui.moderator_path_prefix` (по умолчанию `/mod`). В режиме вебхука
> публичный URL берётся из `webhook.url` автоматически. Затем модераторы запрашивают
> одноразовую ссылку входа в меню бота `/start`. См.
> [Веб‑интерфейс модератора](config_ru.md#веб-интерфейс-модератора-изолированный-ограниченный).

> Примечание: здесь файл конфигурации монтируется **только для чтения**. Правки, сделанные в
> веб‑интерфейсе, нельзя записать обратно в файл, доступный только для чтения. Чтобы сделать
> веб‑интерфейс источником истины, запускайтесь без смонтированного файла конфигурации и
> храните конфигурацию в базе данных - см.
> [следующий раздел](#облачная-установка-вебхук--веб-интерфейс) и
> [Перенос конфигурации…](#перенос-конфигурации-и-базы-данных-между-файлами-и-базой).

## Облачная установка (вебхук + веб‑интерфейс)

Для облачного контейнера (Bunny.net Magic Containers, Fly.io, Render, VPS и т. д.) используйте
**режим вебхука** вместо long‑polling, откройте HTTP‑сервер и отдавайте веб‑интерфейс по HTTPS.

**1. Конфигурация** (файл или переменные окружения):

```yaml
bot_token: "123456:ABC..."

server:
  listen_addr: "0.0.0.0"
  listen_port: 8080

webhook:
  enabled: true
  url: "https://bot.example.com/webhook"   # публичный HTTPS‑URL, ведущий в этот контейнер
  secret_token: "случайный-общий-секрет"   # проверяется в каждом запросе Telegram

web_ui:
  enabled: true
  path_prefix: "/manage-7f3a"   # свой префикс вместо стандартного "/admin"
  password: "надёжный-пароль"
  otp_enabled: true
  # Необязательно: выдать модераторам изолированную ограниченную панель (без конфигурации/логов/системы).
  moderator_path_prefix: "/mod"            # должен отличаться от path_prefix
  public_url: "https://bot.example.com"    # базовый URL для ссылок входа модератора (если опущен, берётся из webhook.url)

admin:
  super_admin_user_id: 123456789
```

> **Доступ модератора (необязательно).** Модераторы могут запросить одноразовую ссылку входа
> в меню бота `/start` (**🛡️ Открыть веб‑интерфейс**). Они попадают на ограниченную панель
> под `moderator_path_prefix` (модерация, сообщения, профили, диагностика только для чтения) —
> конфигурация, логи и системная страница там не обслуживаются. Ссылка строится из
> `web_ui.public_url`; в режиме вебхука он берётся из хоста `webhook.url`, поэтому обычно его
> задавать не нужно. Если публичный URL определить нельзя, кнопка скрыта, а вход модератора отключён.

**2. Требования:**

- **Публичный HTTPS‑эндпоинт**, направляющий на `listen_port` контейнера. Telegram доставляет
  вебхуки только по HTTPS. Настройте TLS на балансировщике или обратном прокси вашей платформы
  или предоставьте сертификат через `server.certificate_path`.
- Бот регистрирует вебхук в Telegram автоматически при запуске, используя `webhook.url`.

**3. Запланированные события в окружениях с вебхуком/serverless.**
В режиме long‑polling бот запускает запланированные задачи (утреннее приветствие, ежедневная
сводка, RSS, профили, очистка) по собственному таймеру. Если ваша платформа приостанавливает
контейнер в период простоя, включите **планирование по вебхуку** и вызывайте триггерный эндпоинт из
внешнего cron / монитора аптайма:

```yaml
scheduled_events:
  webhook_mode: true
  webhook_path: "/trigger-events-9e4b8f2d7c1a4b6f91c0d3e5a8b7c2d4"
  lock_timeout_minutes: 15      # предотвращает дублирующиеся запуски при нескольких экземплярах
```

Используйте длинный случайный `webhook_path`, который сложно угадать, и не публикуйте его в
документации или логах. Затем настройте внешний планировщик периодически обращаться к этому URL
(например, каждые 5–15 минут). Бот выполнит только те задачи, которым пора.

> Оставить **внутренний планировщик** включённым (`webhook_mode: false`) тоже нормально для
> многих облачных конфигураций: даже если платформа приостанавливает/останавливает контейнер в
> простое, входящие вебхук‑события Telegram периодически будят его снова, и при каждом
> пробуждении он выполняет недавно пропущенные события (с учётом
> `scheduled_events.missed_event_max_delay_minutes`).

## Минимальный облачный экземпляр без файла конфигурации

Если ваша платформа затрудняет монтирование `config.yaml`, можно запустить **минимальный
экземпляр** всего с несколькими переменными окружения и удалённой базой данных, а затем
настроить всё остальное через веб‑интерфейс. Это работает потому, что:

- Когда **файл конфигурации не найден**, а настроена **удалённая база данных**, бот загружает
  свою конфигурацию из базы данных.
- Если база данных **пуста**, бот при первом запуске наполняет её из `config.example.yaml`
  (с применением переопределений из переменных окружения).
- Если обязательные значения (токен бота / ID чатов) всё ещё отсутствуют, но **веб‑интерфейс
  включён с пригодной аутентификацией**, бот запускается в ограниченном режиме, чтобы вы могли
  завершить настройку в браузере.

**Необходимые переменные окружения для минимального экземпляра:**

```bash
# Удалённая база данных (чтобы конфигурацию можно было хранить без файла)
DATABASE_URL=libsql://your-db.turso.io
DATABASE_AUTH_TOKEN=your-db-token

# Telegram + как с вами связаться для доставки OTP
BOT_TOKEN=123456:ABC...
ADMIN_SUPER_ADMIN_USER_ID=123456789

# Веб‑интерфейс с пригодной аутентификацией (нужен для запуска без полной конфигурации)
WEB_UI_ENABLED=true
WEB_UI_PASSWORD=надёжный-пароль
WEB_UI_OTP_ENABLED=true

# HTTP‑сервер
SERVER_LISTEN_ADDR=0.0.0.0
SERVER_LISTEN_PORT=8080
```

Веб-интерфейс принимает открытый `WEB_UI_PASSWORD`; при сохранении такой конфигурации в базе
значение преобразуется в `hashed:pbkdf2-sha256:...`. Загрузка базы данных ограничена 128 МБ по
умолчанию; задайте `WEB_UI_MAX_DB_UPLOAD_BYTES`, чтобы изменить лимит.

> «Пригодная аутентификация» означает, что задан **либо** пароль веб‑интерфейса, **либо**
> включён OTP с одновременно заданными `admin.super_admin_user_id` и действующим `bot_token`.
> Без этого бот отказывается запускаться при отсутствии обязательных значений - поэтому он
> никогда не остаётся открытым и ненастроенным.

Запустите контейнер, откройте `https://ваш-хост/admin`, войдите и заполните остальное (ID
чатов модерации, ИИ‑модели, правила, RSS‑ленты). Веб‑интерфейс пишет прямо в базу данных, так
что ваша конфигурация сохраняется между перезапусками и передеплоями без какого‑либо
смонтированного файла.

## Перенос конфигурации и базы данных между файлами и базой

Gennady умеет переносить как **конфигурацию**, так и **хранимые данные** между локальным
файлом и (локальной или удалённой) базой данных. Это мост между «разработкой локально с
файлами» и «работой в облаке с базой данных».

Все эти операции доступны в **веб‑интерфейсе** (разделы «Файлы» / «Диагностика»):

| Действие | Что делает |
|---|---|
| **Скачать конфигурацию** | Экспортировать текущую конфигурацию в файл `config.yaml`. |
| **Загрузить конфигурацию** | Заменить работающую конфигурацию из загруженного `config.yaml`. |
| **Скачать env** | Экспортировать действующую конфигурацию в виде переменных окружения (как `-export-env`). |
| **Скопировать конфигурацию в БД** | Записать текущую файловую конфигурацию в таблицу `config_values` базы данных. |
| **Скачать базу данных** | Экспортировать базу SQLite (опционально без значений конфигурации) в файл. |
| **Загрузить базу данных** | Импортировать файл базы данных, **сливая** данные с текущим хранилищем. |

### Типичные сценарии

**Локальный файл → облачная база.** Настройте локально с `config.yaml`, затем либо:

- в веб‑интерфейсе выберите **Скопировать конфигурацию в БД** (записывает конфигурацию в
  удалённую БД), либо
- запустите бота один раз локально с указанием на удалённую БД, чтобы он её наполнил, либо
- экспортируйте env командой `./gennadium -export-env` и задайте эти переменные в облаке.

**Облачная база → локальный файл.** В веб‑интерфейсе выберите **Скачать конфигурацию**, чтобы
получить `config.yaml`, который можно закоммитить или хранить как резервную копию, и **Скачать
базу данных**, чтобы выгрузить данные в локальный файл SQLite.

### Поведение импорта/слияния данных

Импорт базы данных не разрушает данные там, где это важно. Стратегии по таблицам:

- **Заменяются:** `config_values`, `muted_users`.
- **Сливаются (вставка отсутствующих):** `message_info`, `messages_for_deletion`.
- **Сливаются по естественному ключу (пропуск дублей):** `actions`, `warnings`, `user_names_history`.
- **Сливаются с сохранением новейшего:** `scheduled_events`, `user_profiles`.
- **Сливаются суммированием счётчиков:** `user_daily_activity`.

Это значит, что вы можете выгрузить рабочую базу, поработать локально и залить её обратно, не
потеряв недавнюю активность.

> Флаг командной строки `-export-env` удобен для миграции файловой настройки в конфигурацию на
> основе переменных окружения любой платформы:
> ```bash
> ./gennadium -config config.yaml -export-env > bot.env
> ```

## Настройка удалённой базы данных (Turso / libSQL, BunnyNet)

По умолчанию бот использует **локальный файл SQLite**. Для облачных развёртываний есть **два
варианта**, и между ними можно переключаться позже (см. [Переключение между локальной и удалённой](#переключение-между-локальной-и-удалённой)):

- **Удалённая база libSQL / Turso** - правильный выбор для **эфемерных контейнеров** без
  постоянного хранилища: конфигурация и данные сохраняются между перезапусками и
  передеплоями.
- **Локальный SQLite на постоянном томе** - если ваш облачный провайдер предоставляет
  постоянный диск/том, можно оставить простой локальный файл SQLite (`db/moderation.db`) на нём
  вместо отдельной службы базы данных.

```yaml
database:
  provider: "remote"                       # "local", "remote" или пусто для автоопределения
  url: "libsql://your-db.turso.io"
  auth_token: "your-turso-auth-token"
```

Или через переменные окружения:

```bash
DATABASE_PROVIDER=remote
DATABASE_URL=libsql://your-db.turso.io
DATABASE_AUTH_TOKEN=your-turso-auth-token
```

**Автоопределение провайдера.** Если `provider` пуст или нераспознан, бот определяет
автоматически: если заданы и `url`, и `auth_token`, используется провайдер **remote**, иначе -
**local**.

**Локальный (по умолчанию):**

```yaml
database:
  provider: "local"
  path: "db/moderation.db"
```

**Создание базы Turso:**

```bash
turso db create gennadium
turso db show gennadium --url            # → libsql://...
turso db tokens create gennadium         # → токен аутентификации
```

**BunnyNet.** При работе в BunnyNet Magic Containers переменные окружения платформы
(`BUNNYNET_MC_APPID`, `BUNNYNET_MC_PODID`, `BUNNYNET_MC_REGION`) определяются автоматически и
отображаются в диагностике веб‑интерфейса / логах запуска. Сочетайте контейнер BunnyNet с
удалённой базой Turso и [настройкой минимального экземпляра](#минимальный-облачный-экземпляр-без-файла-конфигурации)
для полностью бесфайлового облачного развёртывания.

### Переключение между локальной и удалённой

Существующее развёртывание можно перенести между **локальным файлом SQLite** и **удалённой**
базой в любом направлении - **локальная → удалённая** или **удалённая → локальная** - без потери
данных:

1. В разделе **Файлы** веб‑интерфейса нажмите **Скачать базу данных** на текущем экземпляре,
   чтобы получить файл SQLite.
2. Направьте новый экземпляр на целевой бэкенд и нажмите **Загрузить базу данных**, чтобы
   импортировать/слить этот файл - подробности о слиянии по таблицам см. в разделе
   [Перенос конфигурации и базы данных…](#перенос-конфигурации-и-базы-данных-между-файлами-и-базой).
3. **Измените переменные окружения**, чтобы бот обращался к новому бэкенду, и перезапустите:
   - На **удалённую**: задайте `DATABASE_PROVIDER=remote`, `DATABASE_URL`, `DATABASE_AUTH_TOKEN`.
   - На **локальную**: задайте `DATABASE_PROVIDER=local` и `DATABASE_PATH`, и удалите переменные
     `DATABASE_URL` / `DATABASE_AUTH_TOKEN`.

> **Не забудьте про переменные окружения.** Переменные окружения переопределяют файл
> конфигурации, поэтому если старые `DATABASE_*` останутся заданными, они победят и бот
> продолжит использовать старый бэкенд.
