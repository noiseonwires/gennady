## Как работает конфигурация

Конфигурация берётся из трёх источников, в порядке возрастания приоритета:

1. **Файл YAML** (`config.yaml`) - обычный источник. Начните с `config.example.yaml`.
2. **База данных** (таблица `config_values`) - используется автоматически, когда файл
   конфигурации не найден и настроена удалённая база данных. Редактируется через веб‑интерфейс.
3. **Переменные окружения** - всегда переопределяют оба источника выше.

**Именование переменных окружения:** возьмите YAML‑путь, замените точки на подчёркивания и
запишите заглавными буквами.

| YAML‑путь | Переменная окружения |
|---|---|
| `bot_token` | `BOT_TOKEN` |
| `admin.chat_id` | `ADMIN_CHAT_ID` |
| `ai.content_moderation.enabled` | `AI_CONTENT_MODERATION_ENABLED` |

Вывести все переменные для вашей текущей конфигурации:

```bash
./gennadium -export-env
```

## Минимальная конфигурация

Минимальная конфигурация, которая запускается (long‑polling, локальный SQLite, без ИИ):

```yaml
bot_token: "123456:ABC-DEF..."
language: "ru"

admin:
  chat_id: -1001234567890        # чат, получающий уведомления модерации

moderation:
  chat_id: -1009876543210        # чат для модерации (один ID или список)

ai:
  enabled: false
```

## База данных

```yaml
database:
  provider: "local"              # "local", "remote" или пусто для автоопределения
  path: "db/moderation.db"       # локальный файл SQLite (provider: local)
  # url: "libsql://your-db.turso.io"   # удалённая БД (provider: remote)
  # auth_token: ""                     # токен удалённой БД
```

Автоопределение: если `provider` пуст/нераспознан и заданы и `url`, и `auth_token`, бот считает
БД **удалённой**, иначе - **локальной**. Настройку Turso см. в
[installation_ru.md](installation_ru.md#настройка-удалённой-базы-данных-turso--libsql-bunnynet).
Удалённая база данных - рекомендуемый вариант для эфемерных облачных контейнеров, таких как
Bunny.net Magic Containers.

## Чаты и темы

Gennady может управлять **несколькими чатами одновременно** и внутри каждого чата -
нацеливаться на отдельные **темы форума**. Поэтому большинство настроек, привязанных к чату,
используют единый формат `(чат, тема)`. Прочитайте этот раздел один раз - он относится к
модерации, удалению сообщений, креативным ответам, сводкам и публикации RSS.

### Что такое «тема»?

**Тема** (также *тема форума* или *тред*) - это подканал внутри Telegram‑группы, в которой
включены **Темы** (Настройки группы → *Темы*). У каждой темы есть свой числовой ID. Область,
которую вы видите в обычной (не форумной) группе - или тред по умолчанию в форумной группе -
называется **основной областью**.

- Если в вашей группе **не** используются темы, ID темы вам не нужен: всё находится в основной
  области.
- Если в группе **используются** темы, функцию можно нацелить на одну конкретную тему, только на
  основную область или на весь чат независимо от темы.

### Значения темы

Везде, где ожидается тема, три значения являются особыми. Каждое принимает числовую форму **или**
строковый псевдоним (регистр не важен):

| Значение | Псевдоним | Смысл |
|---|---|---|
| `-1` | `any`  | **Любая тема** - весь чат, включая основную область |
| `0`  | `main` | **Только основная область** - часть без треда форума |
| `N`  | -      | **Конкретная** тема форума, где `N` - её числовой ID |

`topic: any` и `topic: main` полностью эквивалентны `topic: -1` и `topic: 0`. Веб‑интерфейс и
экспортированные конфиги всегда записывают числовую форму, но псевдонимы можно вводить вручную:

```yaml
included_topics:
  - {chat: -1001112223334, topic: any}    # то же, что topic: -1
  - {chat: -1005556667778, topic: main}   # то же, что topic: 0
```

### Как узнать ID темы

Самый простой способ: откройте тему в **Telegram** (десктоп или веб), нажмите правой кнопкой на
любое сообщение в ней и выберите **Скопировать ссылку на сообщение**. Ссылка выглядит как
`https://t.me/c/1234567890/<ID_ТЕМЫ>/<ID_СООБЩЕНИЯ>` - **среднее** число и есть ID темы.

Либо ненадолго включите отладочное логирование:

1. Задайте `debug.debug_telegram: true` (или переменную `DEBUG_TELEGRAM=true`).
2. Напишите или ответьте в нужной теме - бот залогирует каждое входящее обновление с `chat_id` и
   ID сообщения темы/треда.
3. По окончании выключите отладочное логирование.

### Формат списка `(чат, тема)`

Список `(чат, тема)` записывается объектами. **Оба поля `chat` и `topic` обязательны** в форме
объекта:

```yaml
excluded_topics:
  - {chat: -1001112223334, topic: 42}    # одна конкретная тема
  - {chat: -1005556667778, topic: -1}    # весь чат
  - {chat: -1009998887776, topic: 0}     # только основная область
```

Если вы модерируете **ровно один** чат, можно использовать сокращение - голый ID темы, и бот
сам подставит чат:

```yaml
excluded_topics: [42, 43]                # допустимо только при одном чате модерации
```

> При нескольких чатах модерации сокращение из голых чисел отклоняется при загрузке - указывайте
> явно и `chat`, и `topic`.

### Включённые и исключённые

Область действия функции вычисляется как **включённые минус исключённые**:

- `included_topics` - где функция активна. **Пустой список означает «каждый чат модерации, любая
  тема»** (обычное значение по умолчанию).
- `excluded_topics` - исключения, которые имеют приоритет над включённым набором.

Сообщение в `(чат, тема)` попадает в область, когда его чат является чатом модерации **и** оно
соответствует `included_topics` (или этот список пуст) **и** оно **не** соответствует
`excluded_topics`. Запись с `topic: -1` соответствует каждой теме в этом чате.

Частые шаблоны:

```yaml
# Везде в каждом чате модерации (по умолчанию)
included_topics: []

# Только один чат, только его основная область
included_topics: [{chat: -1001112223334, topic: 0}]

# Весь чат, кроме одной шумной темы
included_topics: [{chat: -1001112223334, topic: -1}]
excluded_topics: [{chat: -1001112223334, topic: 555}]
```

### Куда бот публикует (`post_to`)

Функции, публикующие сообщения от имени бота (утреннее приветствие, ежедневная сводка, RSS‑ленты),
используют список `post_to` с тем же форматом `(чат, тема)`. **Пустой `post_to` публикует в
основную область каждого чата модерации.** Используйте `topic: 0`, чтобы принудительно выбрать
основную область, или `topic: N`, чтобы публиковать в конкретную тему. (`topic: -1` как
получатель не имеет смысла, не используйте его здесь.)

## Админ и модерация

```yaml
admin:
  chat_id: -1001234567890        # куда отправляются уведомления модерации и клавиатуры действий
  reply_message_ids: [42]        # для форум‑чатов: ID сообщений тем для публикации уведомлений
  super_admin_user_id: 123456789 # ваш Telegram user ID (тихие команды, OTP, отладочные ЛС)
  notify_super_admin: false      # также присылать уведомления модерации в ЛС супер‑админу
  notify_startup: false          # ЛС супер‑админу при запуске/перезапуске и потере/восстановлении связи с удалённой БД
  whitelist_user_ids: [777000]   # пользователи, обходящие фильтры и имеющие админ‑доступ; 777000 = служебный аккаунт Telegram (оставьте в белом списке)

moderation:
  chat_id: -1009876543210        # один ID или список: [-100111, -100222]
  excluded_topics: []            # пары (чат, тема), исключённые из анализа контента
  mute_across_all_chats: false   # мут применяется сразу во всех модерируемых чатах
```

`moderation.chat_id` принимает либо одно число, либо список:

```yaml
moderation:
  chat_id: [-1001112223334, -1005556667778]
```

`excluded_topics` использует формат `(чат, тема)` из раздела [Чаты и темы](#чаты-и-темы).
Оставьте пустым, чтобы анализировать всё; добавьте запись вида `{chat: -100111, topic: 42}`,
чтобы пропустить тему, или `{chat: -100111, topic: -1}`, чтобы пропустить весь чат.

## Удаление сообщений

Автоматическое удаление сообщений, при желании ограниченное определёнными темами. Полезно,
чтобы держать тему с объявлениями или медиа чистой.

```yaml
message_deletion:
  enabled: true
  included_topics: []            # где работает удаление (пусто = каждый чат, любая тема)
  excluded_topics: []            # исключения, которые никогда не удаляются
  excluded_user_ids: []          # пользователи, сообщения которых никогда не удаляются
  chat_deletion_retention_hours: 3   # удалять сообщения старше N часов
  cleanup_interval_hours: 3          # запускать очистку удаления каждые N часов
```

Примеры:

```yaml
# Удалять везде (по умолчанию)
included_topics: []

# Удалять только в основной области одного чата
included_topics: [{chat: -1009876543210, topic: 0}]

# Удалять во всём чате, но никогда в его «закреплённой» теме
included_topics: [{chat: -1009876543210, topic: -1}]
excluded_topics: [{chat: -1009876543210, topic: 777}]
```

> Отдельного переключателя «также удалять в основном чате» больше нет. Основная область - это
> просто `topic: 0`: включите её (или используйте `topic: -1` для всего чата), когда хотите её
> охватить.

## Очистка базы данных

Периодическая чистка хранимых записей, чтобы база оставалась небольшой.

```yaml
database_cleanup:
  cleanup_interval_hours: 24     # запускать очистку каждые N часов (0 = выключено)
  message_retention_hours: 168   # хранить message_info 7 дней
  warning_retention_hours: 168
  action_retention_hours: 168
  preserve_warned_muted_messages: false  # хранить сообщения, вызвавшие предупреждение/активный мут
```

## Сервер, вебхук и веб‑интерфейс

Общие настройки HTTP‑сервера, используемые и вебхуком, и веб‑интерфейсом:

```yaml
server:
  listen_addr: "0.0.0.0"
  listen_port: 8080
  # certificate_path: "/path/to/cert.pem"   # TLS‑сертификат для самоподписанных схем
```

**Вебхук** (облако/serverless; по умолчанию используется long‑polling):

```yaml
webhook:
  enabled: false
  url: "https://bot.example.com/webhook"  # публичный HTTPS‑URL
  secret_token: ""                        # общий секрет, проверяемый в каждом запросе
  debug: false                            # подробно логировать тела запросов
```

**Веб‑интерфейс** (панель администратора):

```yaml
web_ui:
  enabled: false
  path_prefix: "/admin"          # префикс URL - измените на свой (см. примечание ниже)
  password: ""                   # пароль администратора или hashed:pbkdf2-sha256:... (пусто = отключено)
  otp_enabled: true              # требовать Telegram‑OTP как 2FA (нужен super_admin_user_id)
  moderator_path_prefix: "/mod"  # префикс URL для изолированного ограниченного UI модератора (должен отличаться от path_prefix)
  public_url: ""                 # базовый URL без префикса (https://bot.example.com); при пустом значении берётся хост вебхука
```

Когда конфигурация хранится в базе данных, `web_ui.password` сохраняется в помеченном формате
`hashed:pbkdf2-sha256:...`. YAML‑конфиги могут использовать как открытый пароль, так и такой
хеш, поэтому экспорт конфигурации из БД в YAML не теряет пароль.

> **Измените префикс пути.** Не оставляйте `path_prefix` со стандартным значением `/admin` на
> публично доступном экземпляре - задайте свой трудноугадываемый вариант (например,
> `/manage-7f3a`), чтобы панель не обнаруживалась тривиально автоматическими сканерами.

> Чтобы запустить бота без полной конфигурации и завершить настройку в браузере, у
> веб‑интерфейса должна быть **пригодная аутентификация**: задан пароль *или* включён OTP с
> одновременно заданными `admin.super_admin_user_id` и действующим `bot_token`. См.
> [installation_ru.md](installation_ru.md#минимальный-облачный-экземпляр-без-файла-конфигурации).

### Веб‑интерфейс модератора (изолированный, ограниченный)

Панель супер‑администратора выше — это полный UI. Модераторам (любому, кто является
администратором в чате модерации/администратора) можно выдать **отдельный ограниченный**
веб‑интерфейс, не раскрывая учётные данные супер‑администратора:

- Он обслуживается по собственному `moderator_path_prefix` (по умолчанию `/mod`, должен
  отличаться от `path_prefix`). Доступны только модерация, сообщения, профили и страница
  диагностики **только для чтения** — конфигурация, логи, системная страница и все
  действия тестирования/отладки под этим префиксом отсутствуют (возвращают 404), поэтому к ним
  нельзя обратиться даже по URL.
- Доступ выдаётся по запросу: модератор открывает бота, нажимает **🛡️ Открыть веб‑интерфейс**
  в меню `/start`, и бот присылает **одноразовую ссылку входа и отдельный OTP**. Открытие
  ссылки и ввод соответствующего OTP создаёт сессию с правами модератора. Ссылка одноразовая,
  истекает через 5 минут и бесполезна без OTP.
- `public_url` — внешне доступный базовый URL для построения этой ссылки
  (например, `https://bot.example.com`), **без** префикса пути. Если он пуст, берётся
  схема + хост из `webhook.url` (удобно для вебхук-развёртываний). Если не задано ни то,
  ни другое, кнопка «Открыть веб‑интерфейс» скрывается, а вход модератора недоступен.

Сессии модератора ограничены ролью: токен модератора отклоняется эндпоинтами
супер‑администратора, а его cookie ограничен префиксом модератора.

## Запланированные события

Управляет тем, как запускаются ежедневные/запланированные задачи (утреннее приветствие,
ежедневная сводка, RSS, профили).

```yaml
scheduled_events:
  missed_event_max_delay_minutes: 60   # запускать пропущенное событие, только если опоздание ≤ N минут
  webhook_mode: false                  # запускать задачи только при триггере через вебхук
  webhook_path: "/trigger-events-CHANGE-ME-LONG-RANDOM"  # путь триггерного эндпоинта
  lock_timeout_minutes: 15             # переустановить устаревшую блокировку через N минут (безопасно при нескольких экземплярах)
```

В режиме `webhook_mode` внешний планировщик должен периодически вызывать `webhook_path`; бот
выполняет только те задачи, которым пора. Для публичных развёртываний используйте длинный
случайный путь, который сложно угадать. См.
[installation_ru.md](installation_ru.md#облачная-установка-вебхук--веб-интерфейс).

## ИИ‑модели

`light_model` и `full_model` принимают каждый **список** эндпоинтов с автоматическим
переключением при сбое (бот переходит к следующей записи при ошибке). Используйте более
дешёвую/быструю модель для массовых задач и более сильную там, где важно качество.

```yaml
ai:
  enabled: true
  chat_rules: "Будьте вежливы. Без спама, без NSFW, только русский/английский."

  # Сохранять покарточные реакции эмодзи→счётчик и подмешивать их в AI-контекст
  # для дневных сводок, креативных ответов и профилей. Агрегированные
  # счётчики работают в любом чате; пособытийные требуют прав админа.
  track_reactions: false

  light_model:
    - provider: "azure"          # "azure" или "openai" (определяется автоматически, если опущено)
      endpoint: "https://YOUR.openai.azure.com/"
      api_key: "KEY"
      deployment_name: "gpt-4o-mini"
      temperature: 0.5
      omit_max_tokens: false     # true для моделей, отвергающих max_tokens (некоторые gpt-5)

  full_model:
    - provider: "azure"
      endpoint: "https://PRIMARY.openai.azure.com/"
      api_key: "KEY"
      deployment_name: "gpt-4o"
      temperature: 0.7
    - provider: "openai"         # запасной вариант на OpenAI‑совместимом шлюзе
      endpoint: "https://api.openai.com/v1"
      api_key: "sk-..."
      deployment_name: "gpt-4o"
      temperature: 0.7
```

- **`provider: azure`** - `endpoint` - базовый URL ресурса, `deployment_name` -
  Azure‑деплоймент, авторизация через заголовок `api-key`.
- **`provider: openai`** - работает с OpenAI и любым OpenAI‑совместимым шлюзом (OpenRouter,
  Groq, LiteLLM, локальный Ollama, …). `endpoint` - базовый URL API, `deployment_name` -
  идентификатор модели, авторизация через Bearer‑токен.

У большинства функций ИИ есть переключатель `use_full_model` для выбора используемой модели.

## Модерация контента ИИ

Сердце бота: классифицировать сообщения настраиваемым промптом, затем направить каждый вердикт
в действие.

```yaml
ai:
  content_moderation:
    enabled: true
    skip_admin_users: true
    complaint_manual_moderation: true   # обработка жалоб «ответ + @упоминание бота» (см. ниже)
    default_mute_minutes: 60     # длительность для действия "mute" (0 = навсегда)
    reply_context_max_chars: 500 # лимит длины цитаты {{reply_to}} (0 = без лимита)

    # Более строгие правила для новых пользователей (первое сообщение в окне new_user_window_hours).
    new_user_window_hours: 24    # сколько пользователь считается «новым»
    new_user_rules: ""           # подставляется через {{new_user_rules}}, только для новых

    # Перепроверять первые N сообщений пользователя полной моделью, даже если
    # лёгкая модель ничего не нашла (ловит неочевидный спам новых участников).
    # Считается отдельно по каждому пользователю и чату. 0 = отключено.
    full_model_first_messages: 0

    # Сопоставление подстроки вердикта модели с действием.
    # Правила выполняются по порядку; срабатывает каждое подходящее правило (действия можно совмещать).
    rules:
      - trigger: "spam"
        action: delete
        description: "Спам / промо‑ссылки"
        notify_admin: true
      - trigger: "profanity"
        action: warn
        notify_admin: true
      - trigger: "nsfw"
        action: mute
        notify_admin: true
      - trigger: "unsure"
        action: report          # переслать в админ‑чат для решения человеком
        notify_admin: true

    prompt:
      system: |
        Ты - классификатор модерации контента для чата. Классифицируй сообщение по одному или
        нескольким ярлыкам нарушений (в нижнем регистре, через запятую): spam, profanity, nsfw,
        unsure, ok. Отвечай только ярлыками. Правила чата: {{chat_rules}}
      user: |
        {{user_profile}}
        {{new_user_rules}}
        {{reply_to}}
        Сообщение: «{{message}}»
        Ответ (только ярлыки):

    warning_prompt:
      system: |
        Ты - модератор чата. Напиши короткое персонализированное предупреждение (1‑2 предложения)
        для пользователя, объясняющее проблему. Добавь эмодзи ⚠️. Правила чата: {{chat_rules}}
      user: "Напиши предупреждение для {{username}}.{{user_message}}"
```

**Действия:**

| Действие | Эффект |
|---|---|
| `delete` | Удалить нарушающее сообщение. |
| `warn` | Выдать сгенерированное ИИ предупреждение (записывается по пользователю). |
| `mute` | Замутить пользователя на `default_mute_minutes` (`0` = навсегда). |
| `report` | Переслать в админ‑чат с клавиатурой действий для ручного решения. |

Чтобы адаптировать бота для другого сообщества, измените `chat_rules`, ярлыки в `prompt` и
сопоставление `rules` - изменения кода не нужны.

### Правила для новых пользователей

Пользователь считается **новым** в течение `new_user_window_hours` (по умолчанию 24) после его
первого замеченного сообщения. Пока он новый, бот добавляет в контекст модерации заметную отметку
«новый пользователь» и разворачивает `{{new_user_rules}}` в текст `new_user_rules` (по умолчанию
пусто). Используйте это, чтобы применять более строгие правила к первым постам, не затрагивая
доверенных завсегдатаев:

```yaml
ai:
  content_moderation:
    new_user_window_hours: 24
    new_user_rules: |-
      Это новый пользователь. Считай любую ссылку, промо или предложение «заработка» спамом.
    prompt:
      user: |
        {{user_profile}}
        {{new_user_rules}}
        Сообщение: «{{message}}»
```

### Жалобы пользователей (ответ + @упоминание бота)

Когда участник **отвечает на сообщение и упоминает бота** в чате модерации, бот воспринимает это
как жалобу. Сначала он заново прогоняет ИИ-модерацию по **каждой отдельной настроенной модели**
(как действие веб-интерфейса **«Проверить снова»**) и действует автоматически, если хотя бы одна
модель нашла нарушение. И только когда **все** модели сочли сообщение чистым, `complaint_manual_moderation`
решает, что делать дальше:

- `true` (по умолчанию) - отправить админам карточку решения для ручной обработки.
- `false` - завершить жалобу без уведомления админов.

## Модерация изображений (зрение и OCR)

Отлавливайте нарушения, спрятанные в изображениях, извлекая их текст/содержимое и пропуская его
через тот же конвейер модерации.

```yaml
ai:
  content_moderation:
    # Azure Vision - описание изображений + OCR
    vision_enabled: true
    vision_endpoint: "https://YOUR.cognitiveservices.azure.com/"
    vision_api_key: "KEY"

    # Azure Content Safety - опциональная оценка безопасности
    content_safety_enabled: false
    content_safety_endpoint: "https://YOUR-SAFETY.cognitiveservices.azure.com/"
    content_safety_api_key: "KEY"

    # Проверять ВЕСЬ публичный профиль нового участника при первом
    # сообщении: имя, био и фото профиля, плюс привязанный личный канал
    # (название, описание, фото). Фото сначала проверяются через Content Safety,
    # и только если он недоступен или дал сбой - описываются через Vision /
    # OCR.space; весь текст оценивает new_user_profile_prompt.
    # Работает БЕЗ content_safety_enabled. Промпт ОБЯЗАН отвечать ровно
    # CLEAN для нормальных профилей; любой другой ответ становится отметкой.
    new_user_profile_check_enabled: false
    # Оценивать собранный текст профиля ПОЛНОЙ моделью вместо лёгкой - лучше
    # распознаёт неочевидные признаки спама/скама/рекламы, но дороже за вызов.
    # Влияет только на текстовый вердикт AI; проверка фото не меняется. По умолчанию: false.
    new_user_profile_use_full_model: false
    new_user_profile_prompt:
      user: |
        Проанализируй следующий профиль:

        {{profile_text}}

    # Облачный OCR API OCR.space - без своего сервера, есть бесплатный тариф
    ocrspace_enabled: false
    ocrspace_api_key: "YOUR_OCRSPACE_KEY"  # тестовый ключ: helloworld
    ocrspace_url: "https://api.ocr.space/parse/image"
    ocrspace_language: "eng"               # 3-буквенный код; движки 2/3 принимают "auto"
    ocrspace_engine: 2                     # 1=по умолчанию, 2=универсальный, 3=макс. точность
```

> **[OCR.space](https://ocr.space/ocrapi)** - облачная альтернатива: задайте `ocrspace_enabled: true`
> и `ocrspace_api_key`, чтобы использовать облачный OCR без своего сервера (бесплатный тариф: 25 000
> запросов/месяц, лимит файла 1 МБ). Порядок резервирования: **Azure Vision → OCR.space**. Движок
> `2` хорошо подходит для мемов и шумного фона; движок `3` - самый точный.

## Креативные ответы

Эпизодические остроумные, учитывающие контекст ответы ИИ - с ограничением частоты, чтобы бот
оставался обаятельным, а не назойливым.

```yaml
ai:
  creative_replies:
    enabled: true
    use_full_model: true
    max_messages: 3              # максимум ответов за временное окно
    time_window: 3              # длина окна в часах
    included_topics: []          # где разрешены ответы (пусто = каждый чат, любая тема)
    excluded_topics: []          # темы, где ответы подавляются
    follow_up_only_same_user: false
    prompt:
      system: "Ты - остроумный участник чата. Отвечай естественно и коротко."
      user: |
        Ответь на: "{{message}}"
        {{context}}
```

Оба списка используют формат `(чат, тема)` из раздела [Чаты и темы](#чаты-и-темы). Чтобы
разрешить ответы только в одной теме, задайте `included_topics: [{chat: -100111, topic: 42}]`.

## Утреннее приветствие

Ежедневное приветствие, которое может включать погоду, праздники и исторические события.

```yaml
ai:
  morning_greeting:
    enabled: true
    use_ai: true                 # false = только данные (погода/праздники/события, без текста ИИ)
    use_full_model: true
    time: "08:00"                # HH:MM, местное время (задайте переменную TZ)
    post_to: []                  # куда публиковать (пусто = каждый чат модерации, основная область)
    prompt:
      system: "Ты пишешь короткое дружелюбное утреннее приветствие для чата."
      user: "Создай приветствие. Сегодня {{weekday}} {{date}}{{weather}}{{holidays}}{{events}}"

  external_data:
    weather_latitude: 50.088     # по умолчанию Прага
    weather_longitude: 14.4208
    holidays_country: "CZ"       # ISO‑код страны (Open Holidays API)
    wikipedia_language: "cs"     # язык событий «в этот день» из Wikipedia
    translate_wikipedia: true    # переводить события через ИИ
```

Погода (Open‑Meteo), праздники (OpenHolidays API) и события Wikipedia бесплатны и не требуют
API‑ключа.

## Сводки сообщений и ссылок

**Сводки сообщений** - автоматическое резюмирование длинных сообщений:

```yaml
ai:
  message_summaries:
    enabled: true
    use_full_model: false
    light_model_threshold: 4096  # принудительно лёгкая модель свыше N символов (0 = выкл.)
    min_length: 1000             # резюмировать только сообщения длиннее этого
    included_topics: []          # где работает резюмирование (пусто = каждый чат, любая тема)
    excluded_topics: []          # темы, которые пропускаются
    excluded_user_ids: []
    prompt:
      system: "Ты кратко резюмируешь сообщения."
      user: |
        Резюмируй в 1‑2 предложениях:
        {{message}}
```

**Сводки ссылок** - загрузить присланные ссылки и опубликовать краткий пересказ. Использует цепочку
запасных провайдеров извлечения (Diffbot → ExtractorAPI → Cloudflare Browser Rendering →
ручной).

```yaml
ai:
  link_summaries:
    enabled: true
    use_full_model: false
    light_model_threshold: 8192
    excluded_domains: ["t.me", "telegram.org"]
    excluded_extensions: [".pdf", ".zip", ".mp4", ".exe"]
    excluded_user_ids: []
    included_topics: []          # где работают сводки ссылок (пусто = каждый чат, любая тема)
    excluded_topics: []          # темы, которые пропускаются
    extractor_api_key: ""        # ExtractorAPI (опционально)
    diffbot_api_key: ""          # Diffbot (опционально)
    # cloudflare_account_id: ""  # Cloudflare Browser Rendering (опционально, для страниц с JS)
    # cloudflare_api_token: ""
    # cookies: "name=value"      # куки для извлечения (обход баннеров согласия)
    # user_agent: "Mozilla/5.0 ..."
    content_language: "ru"
    max_extracted_content_length: 4096
    max_download_size_bytes: 1048576   # 1 МБ
    min_summary_length: 0        # отбрасывать резюме короче N символов (0 = выкл.)
    prompt:
      system: "Ты кратко резюмируешь веб‑страницы."
      user: |
        Заголовок: {{title}}
        URL: {{url}}
        Содержимое{{truncated_suffix}}:
        {{content}}
```

ai:

    use_full_model: true

## Профили пользователей

Две независимые системы:

**ИИ‑профили** (`ai.user_profiles`) - ежедневно формируемые поведенческие профили с оценкой
репутации, подставляемые в модерацию через `{{user_profile}}`. Есть и компактная, экономящая
токены альтернатива - `{{user_reputation}}`, которая содержит только оценку репутации плюс
отметку о том, что фото профиля пользователя помечено системой контент‑безопасности:

```yaml
ai:
  user_profiles:
    enabled: false
    time: "03:00"
    skip_forever_muted_users: false
    prompt:
      system: |
        Ты - аналитик поведения. Напиши КОРОТКИЙ профиль (2‑3 предложения) и закончи строкой:
        "Reputation: good" / "neutral" / "bad".
      user: |
        Составь профиль @{{username}} на основе:
        {{messages}}
    update_prompt:
      system: "Обнови существующий профиль на основе новой активности. Держи коротко."
      user: |
        Текущий профиль:
        {{existing_profile}}
        Новая активность:
        {{messages}}
```

**Отслеживание** (`user_profiles`) - не‑ИИ отслеживание истории имён/отображаемых имён, отметок
времени первого появления, ежедневных счётчиков активности и оповещений о подмене:

```yaml
user_profiles:
  enabled: false
  disable_username_reuse_alerts: false   # подавить оповещения «ник переиспользован новым аккаунтом»
```

## Реакции и отладка

У эмодзи‑реакций есть разумные значения по умолчанию; переопределяйте только при желании:

```yaml
reactions:
  suspicious_message: "🤔"
  bad_message: "🍌"
  content_filter: "🥴"
  creative_reply_limit: "🥱"
  extracting_link: "✍"
  extract_link_failed: "🌚"
  user_muted: "🤮"
  report_acknowledged: "👌"
  creative_reply_error: "😐"
```

> Используйте только те эмодзи, которые Telegram разрешает **в качестве реакций на сообщения**.
> Telegram принимает фиксированный набор эмодзи‑реакций; если задать эмодзи вне этого набора, бот не
> сможет поставить реакцию. Придерживайтесь значений по умолчанию или выбирайте замены из списка
> поддерживаемых Telegram реакций.

Отладочное логирование:

```yaml
debug:
  debug_telegram: false          # логировать обновления, команды, коллбэки (полезно для поиска ID чатов)
  debug_external_apis: false     # логировать все запросы/ответы внешних API
  debug_api_errors: false        # логировать только неудачные запросы
  dump_moderation_messages: false
  dump_admin_messages: false
  message_dump_path: "./logs"
  send_to_super_admin: false     # присылать включённые отладочные логи в ЛС супер‑админу
```
